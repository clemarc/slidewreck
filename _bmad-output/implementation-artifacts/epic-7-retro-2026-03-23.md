## Successes

1. **Spike-to-implementation handoff is a proven, repeatable pattern** — 3rd consecutive epic where the spike correctly scoped the work. Story 7.1 flagged 3 risks (attribute names, TraceQL metrics availability, cost calculation limitations); Story 7.2 systematically resolved all 3.

2. **Attribute name verification from source code** — Rather than guessing TraceQL attribute names, Charlie verified them directly from `@mastra/otel-exporter` source. All dashboard queries use confirmed GenAI semantic conventions (`gen_ai.operation_name`, `gen_ai.agent.name`, `gen_ai.request.model`, `gen_ai.usage.*`).

3. **Dashboard-as-code with `allowUiUpdates: false`** — Clean infrastructure decision. Dashboards are version-controlled, reproducible, and can't drift from manual UI edits.

4. **12 structural validation tests as safety net** — Valid JSON, required fields, no duplicate panel IDs, correct datasource references, provisioning config validation. Catches schema breakage before container startup.

5. **Leanest epic yet** — 2 stories, zero blockers, zero debt, zero architectural pivots. Pure execution.

---

## Challenges

1. **E2E validation gap** — Dashboard queries and provisioning are only fully testable against a live LGTM stack with real TalkForge traces. Structural tests validate JSON correctness but cannot verify that TraceQL queries return data. The OTEL collector URL config and dashboard loading issues required manual back-and-forth to diagnose.

2. **Slow feedback loop for infrastructure wiring** — During development, dashboard JSON and TraceQL queries are written essentially blind. Problems only surface when the full stack (Mastra → OtelExporter → LGTM → Grafana → Tempo) is running with actual trace data.

---

## Key Insights

1. **Config-heavy epics have a distinct quality profile** — Fewer logic bugs but more wiring issues that only appear in E2E testing. Both Epic 6 (OTEL integration) and Epic 7 (Grafana dashboards) exhibited this pattern. Unit/structural tests prove correctness; wiring between systems requires live verification.

2. **E2E testing gap is acceptable for a learning project** — The team knows exactly what production would need (integration smoke tests with Docker containers, trace propagation, Grafana API verification). Building that infrastructure would exceed the epic's scope and learning ROI. Accepted as known cost.

3. **"Unknown-unknowns in third-party integration are expected cost" — validated again** — First identified in Epic 6 retro, reconfirmed in Epic 7. The OTEL collector URL config issue was not anticipatable during the spike. Fast diagnosis beats heavy prevention for this project.

4. **Spike investment continues to compound** — This was the second-lightest non-trivial epic (after Epic 6). Spike mapped the full surface, prototype was built ahead of schedule, implementation was polish and verification.
