## Challenges

1. **Story 10.3 acceptance criteria didn't cross-reference backend resume schemas** — The generic `{decision, feedback}` form only worked for 2 of 5 gates. `collect-references` expects `{materials: [...]}` with no decision field at all. `architect-structure` needs option selection. `review-slides` needs optional DeckSpec editing. Required a mid-sprint change proposal to add Story 10.5.

2. **`resume-async` query param convention not caught by spike** — The endpoint uses `?runId=...` as a query parameter rather than a path parameter. Required a hotfix (commit `4080fa8d`). Spikes are better but endpoint calling conventions still slip through.

3. **Dev Agent Records empty for Stories 10-1 through 10-4** — No completion notes, no agent model, no file lists, no test counts recorded. Only Story 10-5 has a full record. Lost implementation context for this retrospective.

---

## Key Insights

1. **Stories touching suspend/resume need schema enumeration during planning** — For each gate, cross-reference the backend Zod resume schema and verify the UI story covers that shape. This would have caught the Story 10.3 gap upfront.

2. **The change proposal process is a healthy correction mechanism** — Mid-sprint course correction is not a failure signal. The formal process (document the gap, scope the fix, get approval) kept quality high without derailing velocity.

3. **Pattern reuse is the architecture paying dividends** — When component patterns are consistent (dispatcher → per-type renderer), new work follows established tracks. The gate-controls dispatcher was implemented quickly because gate-content showed the way.

4. **Dev Agent Records are only valuable if enforced** — Without completion data, retrospectives lose granularity. Enforcement in code review is the fix.
