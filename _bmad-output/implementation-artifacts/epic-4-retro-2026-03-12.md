## Successes

1. **100% completion with all FRs/NFRs delivered** — Complete eval pipeline: scorers, scorecard, storage, trends, standalone CLI. Clean sweep of business requirements.

2. **API verification spike prevented major architectural pivot** — Spike caught that `runEvals` requires a target Agent/Workflow (dataset-level evaluation), not suitable for scoring existing output. Team designed the simpler inline `runEvalSuite()` approach before development started.

3. **Pattern reuse compounded across stories** — Story 4-1 established the `createScorer` builder pattern. Stories 4-2 through 4-4 followed the recipe mechanically. Fastest epic delivered (~1 day implementation).

4. **12/14 Epic 3 action items completed** — Massive turnaround from 0/6 in prior retro. One-epic-per-sprint cadence with action item gap proved its value.

5. **Code review findings self-corrected by mid-epic** — 2 findings in early stories (rating labels, registration pattern), zero findings in Stories 4-3 and 4-4. Lessons applied forward immediately.

6. **52 new tests with TDD discipline** — 194 → 246 across 4 stories. Never a broken build. Heuristic scorer tests (pacing distribution) are deterministic and test real logic.

7. **Dependency injection for testability** — Story 4-4 passed ScoresStorage as parameter, no Mastra instance or Docker needed in tests. Clean architecture pattern.

---

## Challenges

1. **`getSteps()` bug in `@mastra/core@1.8.0`** — Crashes for prompt-based steps (definition is undefined). Workaround: source inspection via `readFileSync` in tests. Appeared in Stories 4-1 and 4-2. Framework limitation, documented in Known Limitations.

2. **Scorer registration pattern confusion** — Type definitions say `MastraScorerEntry = { scorer }` wrapping, but Mastra constructor expects direct instances (`Record<string, MastraScorer>`). Story 4-1 shipped with wrong pattern, caught and fixed in Story 4-2 code review (retroactively fixed all 4 scorers).

3. **`runEvals` API mismatch** — `runEvals` from `@mastra/core/evals` requires a target Agent/Workflow for dataset-level evaluation. Not suitable for scoring existing output. Caught by spike, but still required architectural redesign of Story 4-3's approach.

4. **Type definition / runtime behavior divergence** — Multiple instances where `@mastra/core` TypeScript types didn't match actual runtime behavior. Early adopter cost of a young framework.

---

## Key Insights

1. **API verification spikes before sprints eliminate architectural surprises** — Proved again: spike caught `runEvals` mismatch. Two epics in a row where the spike prevented significant rework.

2. **Pattern reuse compounds exponentially** — Investment in clean infrastructure in Story 4-1 paid compound returns in 4-2, 4-3, and 4-4. Later stories were almost mechanical.

3. **One-epic-per-sprint cadence with action item gap works** — 12/14 completion vs 0/6 previously. The structural gap between epics is the key enabler.

4. **Early adopter framework friction is manageable when documented systematically** — `getSteps()` bug, registration pattern, `runEvals` mismatch — all documented in Known Limitations. Team treats workarounds as temporary, monitors for framework fixes.

5. **Sometimes the custom path is simpler than the framework's intended path** — Inline `runEvalSuite()` with `Promise.allSettled` is simpler and more flexible than `runEvals`. Don't force-fit framework APIs when a straightforward solution exists.

6. **Reusable workflow patterns are emerging** — Spike cadence, defensive validation, action item tracking, TDD flywheel — these are transferable to other projects. Worth extracting.
