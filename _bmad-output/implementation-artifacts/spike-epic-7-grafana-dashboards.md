## 1. Dashboard Provisioning Pipeline

### Approach: File-Based Provisioning

The `grafana/otel-lgtm` image supports standard Grafana provisioning. Dashboards are version-controlled as JSON files and auto-loaded by the container on startup.

**Directory structure:**

```
grafana/
├── provisioning/
│   └── dashboards/
│       └── dashboards.yaml    # Provisioning config (tells Grafana where to find dashboards)
└── dashboards/
    └── talkforge-pipeline.json  # Dashboard JSON (the actual dashboard)
```

**Provisioning config** (`grafana/provisioning/dashboards/dashboards.yaml`):

```yaml
apiVersion: 1
providers:
  - name: "TalkForge"
    orgId: 1
    folder: "TalkForge"
    type: file
    disableDeletion: false
    updateIntervalSeconds: 10
    options:
      path: /var/lib/grafana/dashboards
      foldersFromFilesStructure: false
```

**Docker Compose volume mounts** (added to `lgtm` service):

```yaml
volumes:
  - ./grafana/provisioning/dashboards:/etc/grafana/provisioning/dashboards
  - ./grafana/dashboards:/var/lib/grafana/dashboards
```

**How it works:**
1. `dashboards.yaml` is mounted to `/etc/grafana/provisioning/dashboards/` — Grafana reads this on startup
2. The config points to `/var/lib/grafana/dashboards/` as the dashboard file location
3. Dashboard JSON files are mounted to `/var/lib/grafana/dashboards/`
4. Grafana polls every 10 seconds (`updateIntervalSeconds: 10`), so changes to JSON files are reflected automatically
5. Dashboards appear in a "TalkForge" folder in the Grafana UI

**Key finding:** The LGTM image already pre-configures Tempo as a datasource with UID `tempo`. No datasource provisioning needed — we reference `{ "type": "tempo", "uid": "tempo" }` in panel targets.

### Datasource Note

The LGTM image comes with pre-configured datasources:
- **Tempo** — traces (primary for our dashboards)
- **Loki** — logs (container logs ingested via ENABLE_LOGS_ALL)
- **Mimir** (Prometheus-compatible) — metrics

For TalkForge, Tempo is the primary datasource. Mimir metrics are not generated by Mastra's OtelExporter (it exports spans, not metrics), so metric-based panels are not feasible without additional configuration.

## 2. Tempo TraceQL Queries

### Span Attribute Mapping (Verified in Story 7.2)

**Updated 2026-03-19:** Attribute names verified by reading `@mastra/otel-exporter` source code. The exporter follows OpenTelemetry GenAI semantic conventions.

| OTEL Attribute | Description | Example Values | Span Types |
|----------------|-------------|---------------|------------|
| `resource.service.name` | Service name from Observability config | `slidewreck` | All |
| `gen_ai.operation_name` | Semantic operation name | `invoke_workflow`, `invoke_agent`, `chat`, `execute_tool`, `workflow_step`, `workflow_wait_event` | All |
| `mastra.span.type` | Raw Mastra SpanType enum | `WORKFLOW_RUN`, `AGENT_RUN`, `MODEL_GENERATION`, `TOOL_CALL` | All |
| `gen_ai.agent.name` | Agent name | `researcher`, `writer`, `designer` | AGENT_RUN, MODEL_GENERATION |
| `gen_ai.agent.id` | Agent ID | `researcher`, `script-writer` | AGENT_RUN, MODEL_GENERATION |
| `gen_ai.tool.name` | Tool name | `web-search`, `word-count-to-time` | TOOL_CALL, MCP_TOOL_CALL |
| `gen_ai.request.model` | LLM model ID | `claude-opus-4-6`, `claude-sonnet-4-5` | MODEL_GENERATION |
| `gen_ai.usage.input_tokens` | Input token count | numeric | MODEL_GENERATION |
| `gen_ai.usage.output_tokens` | Output token count | numeric | MODEL_GENERATION |
| `gen_ai.usage.cached_input_tokens` | Cache hit tokens | numeric | MODEL_GENERATION |
| `gen_ai.usage.reasoning_tokens` | Thinking/reasoning tokens | numeric | MODEL_GENERATION |
| `status` | OTEL status code | `unset`, `ok`, `error` | All |
| `duration` | Span duration | intrinsic | All |

### Candidate TraceQL Query Patterns (Attribute Names Unverified)

**Basic trace search:**
```
{ resource.service.name = "slidewreck" }
```

**Filter by span type:**
```
{ resource.service.name = "slidewreck" && span.type = "AGENT_RUN" }
{ resource.service.name = "slidewreck" && span.type = "MODEL_GENERATION" }
{ resource.service.name = "slidewreck" && span.type = "WORKFLOW_STEP" }
{ resource.service.name = "slidewreck" && span.type = "TOOL_CALL" }
{ resource.service.name = "slidewreck" && span.type = "WORKFLOW_WAIT_EVENT" }
```

**Filter by entity:**
```
{ resource.service.name = "slidewreck" && span.entity.name = "researcher" }
{ resource.service.name = "slidewreck" && span.model.name = "claude-opus-4-6" }
```

**Error traces:**
```
{ resource.service.name = "slidewreck" && status = error }
```

### TraceQL Metrics Queries

TraceQL metrics enable aggregation panels without a metrics backend:

**Count by type:**
```
{ resource.service.name = "slidewreck" && span.type = "WORKFLOW_RUN" } | count() by (span.status)
```

**Duration quantiles:**
```
{ resource.service.name = "slidewreck" && name = "workflow.run" } | quantile_over_time(duration, 0.5, 0.95, 0.99)
```

**Rate:**
```
{ resource.service.name = "slidewreck" && span.type = "AGENT_RUN" } | rate() by (span.entity.name)
```

### Limitations

1. **TraceQL metrics** require Tempo's `metrics-generator` to be enabled. The LGTM image may or may not have this enabled by default. If not, metric panels will fall back to trace search + table views.
2. **Token usage aggregation** depends on whether `span.usage.inputTokens` is exported as a numeric span attribute. If it's in the span's `input`/`output` payload instead, it won't be queryable via TraceQL.
3. **Cost calculation** cannot be done in TraceQL — there's no arithmetic on attributes. Cost estimation would need a separate panel type or a post-processing step.

## 3. Dashboard Panel Specifications

### Dashboard: TalkForge Pipeline (`talkforge-pipeline.json`)

Organized in 5 collapsible row sections:

#### Row 1: Pipeline Overview

| Panel | Type | Query | Notes |
|-------|------|-------|-------|
| Workflow Runs (24h) | Stat | Count WORKFLOW_RUN spans by status | Quick health check |
| Workflow Run Duration | Time series | p50/p95 duration of WORKFLOW_RUN spans | Trend over time |
| Recent Workflow Runs | Table | Search WORKFLOW_RUN spans, limit 20 | Clickable trace links |

#### Row 2: Agent Performance

| Panel | Type | Query | Notes |
|-------|------|-------|-------|
| Agent Run Duration | Time series (bars) | p50 duration of AGENT_RUN by entity.name | Compare agent execution times |
| Tool Call Frequency | Bar chart | Count TOOL_CALL by entity.name | See which tools are used most |

#### Row 3: Token Usage & Cost

| Panel | Type | Query | Notes |
|-------|------|-------|-------|
| Model Call Rate | Time series | Rate of MODEL_GENERATION spans by model.name | Track LLM call frequency per tier (token counts need attribute verification) |
| Model Call Latency | Time series | p50/p95 duration of MODEL_GENERATION by model.name | LLM response time monitoring |

**Cost Tracking Note:** Direct cost calculation from TraceQL is not possible. Two alternatives for Story 7.2:
1. **Documentation panel** with a static cost table (Opus: $15/$75, Sonnet: $3/$15, Haiku: $0.80/$4 per 1M tokens) — users mentally correlate with token usage panel
2. **Grafana transformations** — if token counts are available as numeric attributes, use "Add field from calculation" to multiply by cost rates

#### Row 4: Step-Level Drilldown

| Panel | Type | Query | Notes |
|-------|------|-------|-------|
| Workflow Step Durations | Time series (bars) | p50 duration of WORKFLOW_STEP by entity.name | Identify slow steps |
| Suspend/Resume Wait Times | Time series | p50 duration of WORKFLOW_WAIT_EVENT by entity.name | Human review latency |

#### Row 5: Error Analysis

| Panel | Type | Query | Notes |
|-------|------|-------|-------|
| Errors by Step | Table | Search spans with status=error, limit 50 | Error investigation |

### Panel Count: 10 panels across 5 sections

This is a manageable dashboard that covers all the candidate areas from the epic description without overloading.

## 4. Prototype Validation

### What Was Built

A complete prototype dashboard (`grafana/dashboards/talkforge-pipeline.json`) with:
- 10 panels across 5 collapsible row sections
- TraceQL and TraceQL metrics queries targeting TalkForge span types
- Tempo as the datasource (pre-configured in LGTM)
- Provisioning config to auto-load on container restart

### Provisioning Pipeline

```
grafana/provisioning/dashboards/dashboards.yaml  →  mounted to /etc/grafana/provisioning/dashboards/
grafana/dashboards/talkforge-pipeline.json       →  mounted to /var/lib/grafana/dashboards/
docker-compose.yml                               →  volume mounts added to lgtm service
```

**To validate:** `docker compose down && docker compose up -d`, then navigate to `http://localhost:3200` → Dashboards → TalkForge folder.

### What Needs Verification in Story 7.2

1. **Attribute names** — verify actual OTEL span attribute names by inspecting a real trace in Grafana Explore > Tempo. The Mastra OtelExporter may map attributes differently than assumed.
2. **TraceQL metrics availability** — confirm the LGTM image has `metrics-generator` enabled for `quantile_over_time()`, `count()`, `rate()` queries.
3. **Token usage attributes** — check if `span.usage.inputTokens` / `span.usage.outputTokens` are queryable numeric attributes or embedded in span payload.
4. **Panel rendering** — verify each panel actually renders data (not just empty).

## 6. Risks & Open Questions

1. **Attribute name mismatch** (HIGH) — The OtelExporter may use different attribute names than assumed. Story 7.2 must verify against actual data first.
2. **TraceQL metrics availability** (MEDIUM) — LGTM image may not enable `metrics-generator` by default. Fallback: use trace search + table panels.
3. **Token count accessibility** (MEDIUM) — Token counts may be in span payloads (not queryable) rather than span attributes.
4. **LGTM data retention** (LOW) — Default retention is likely short (hours/days). Acceptable for local dev.
5. **Dashboard version compatibility** (LOW) — `schemaVersion: 39` may not match the Grafana version in the LGTM image. Update if needed.
