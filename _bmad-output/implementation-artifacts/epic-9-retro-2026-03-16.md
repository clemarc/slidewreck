## Challenges

1. **Mastra API response shape mismatch (BLOCKER)** — `start-async` endpoint returns `{ status, result, error }`, NOT `{ runId }`. Architecture.md had the wrong endpoint shape. The spike verified endpoint existence but not response bodies. Result: `/run/undefined` on first real use. Required a post-sprint hotfix to two-step flow: `create-run` (returns `{ runId }`) then `start-async` with runId query param.

2. **Architecture.md API table was wrong** — Listed `POST /api/workflows/{workflowId}` when the actual endpoint is `/workflows/{workflowId}/start-async`. Resume endpoint was also wrong (`/{runId}/resume` vs `/resume-async` with runId in body). The table was never updated after the spike verified the real endpoints.

3. **Mock-only API client testing insufficient** — All 10 client tests passed with mocked fetch responses. None caught the real response shape mismatch because the mocks encoded our assumptions, not reality.

4. **Mastra dev server has Swagger UI and OpenAPI spec — not leveraged** — The dev server exposes machine-readable API documentation that would have given us the exact response schemas. We reverse-engineered bundled source code instead.

---

## Key Insights

1. **Mocked tests verify your assumptions, not reality** — API client stories need at least one integration smoke test against the real server. A single `curl` to `start-async` would have revealed the missing `runId` field.

2. **Use the OpenAPI spec, not source code** — The Mastra dev server exposes Swagger UI and an OpenAPI schema. Fetching the spec is faster, more reliable, and less error-prone than reading bundled JavaScript handler source.

3. **Architecture.md must be updated when assumptions are disproven** — Story 9-2 discovered the real endpoints but didn't update the API table. The wrong information persisted and would mislead Epic 10 development.

4. **Two-step create-run + start-async is the correct Mastra pattern** — This is the canonical flow for getting a run ID before starting execution. Must be encoded in architecture.md.

5. **The spike checklist gap: response shape verification** — Spike checklist expanded in E5 to include payload size and structured output testing. Now needs another expansion: "make a real HTTP request and document the actual response shape."
