## Challenges

1. **SVG diagram inlining broke with real-world data** — The initial implementation assumed `diagram.svgPath` would be set in the DeckSpec JSON. In reality, the workflow saves SVGs to disk but doesn't populate `svgPath` in the JSON. Required a post-sprint fix to add convention-based auto-discovery (`diagrams/diagram-<slideNumber>.svg`).

2. **Marp markdown image syntax fails with long base64 data URIs** — `![diagram](data:image/svg+xml;base64,...)` rendered as literal text in the PDF because the data URI was too long for Marp's markdown parser. Had to switch to HTML `<img>` tags and enable `html: true` in the Marp constructor.

3. **Diagrams on non-diagram layouts were ignored** — The initial `renderContent` switch only called `renderDiagramContent` for `diagram` layout. Slides with `comparison` or `content` layout that also had a `diagram` field got no SVG embedding. Required broadening the diagram inlining to all layouts.

4. **Test fixture didn't catch real-world issues** — The sample-deck.json fixture was too simple (no real SVG files on disk, only 6 slides). The three bugs above were only discovered during end-to-end testing with a real 19-slide presentation from the workflow.

---

## Key Insights

1. **Spike verified API shape but not integration contract** — The spike tested Marp rendering and Puppeteer PDF generation in isolation. It did not test the full pipeline with actual workflow output (DeckSpec JSON without `svgPath`, co-located SVG files). The gap between "API works" and "pipeline works end-to-end" is where bugs live.

2. **Convention-over-configuration works when documented** — Auto-discovering `diagrams/diagram-<N>.svg` by slide number eliminated the need to modify the workflow to set `svgPath`. But this convention must be documented so future changes to the workflow's output structure don't silently break rendering.

3. **HTML mode in Marp is essential for data URIs** — Marp's markdown parser has limitations with very long inline content. Using `html: true` + `<img>` tags is more robust for programmatic content injection. This should be the default for any Marp programmatic usage.

4. **Standalone scripts architecture is excellent for rendering pipelines** — Zero coupling to Mastra framework means tests are fast, dependencies are minimal, and the CLI can be used independently. Good precedent for future utility scripts.
